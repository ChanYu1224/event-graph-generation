## 1. 全体像

本システムは**動画から「誰が何をしたか」を構造化データとして出力する**モデルである。出力はテキストではなく、オブジェクト（物体・人）とイベント（動作）の関係を表す**イベントグラフ**（JSON構造体）。

処理の大まかな流れ:

```
動画 → 映像特徴量抽出 → オブジェクト発見 → イベント予測 → イベントグラフ JSON
```

2つのパイプラインが存在する:

| パイプライン | 映像エンコーダ | オブジェクト検出 | 状態 |
|---|---|---|---|
| **V-JEPA パイプライン** | V-JEPA 2.1 (凍結) | Slot Attention (学習) | メイン |
| **SAM3 パイプライン** | SAM 3 (凍結) | SAM 3 Tracker (凍結) | レガシー |

いずれも最終段の**Event Decoder**（DETR風セット予測モデル）が共通設計であり、固定数のイベントクエリから一括でイベントを予測する。

## 3. V-JEPA パイプライン（メイン）

### 3.1 V-JEPA 特徴量抽出（凍結）

V-JEPA（Video Joint-Embedding Predictive Architecture）は Meta が開発した自己教師あり映像エンコーダである。映像のパッチ列を入力として受け取り、時空間にまたがるトークン列を出力する。

```
入力:  16フレームの映像クリップ (B, 3, 384, 384)
出力:  spatiotemporal tokens (B, 1568, 1024)
       = 8 temporal tokens × 196 spatial tokens × 1024次元
```

| バリアント | パラメータ数 | 隠れ次元 | 用途 |
|---|---|---|---|
| ViT-B | 80M | 768 | 軽量実験 |
| ViT-L | 300M | 1024 | 標準（デフォルト） |
| ViT-g | 1B | 1408 | 高精度 |
| ViT-G | 2B | 1664 | 最高精度 |

V-JEPA は完全に凍結され、学習時にパラメータ更新は行わない。事前に全動画の特徴量を `.pt` ファイルとして抽出・保存しておき、学習時はそれを読み込む。

**なぜ V-JEPA か:**
- 自己教師あり学習済みのため、ラベル不要で汎用的な映像表現を獲得している
- 時空間を統一的にトークン化するため、下流のオブジェクト発見モジュールで時空間推論が可能
- 凍結して使うことで学習コストを大幅に削減（学習対象は後段のみ）

### 3.2 ObjectPoolingModule（Slot Attention）

V-JEPA が出力する 1568 個のトークン（映像パッチの特徴量）を、**K=24 個のオブジェクトスロット**に集約するモジュール。4つのサブモジュールで構成される。

#### 3.2.1 InputProjection

```python
# (B, 1568, 1024) → (B, 1568, 256)
proj = nn.Linear(1024, 256)  # 次元圧縮
+ 時間位置エンコーディング (8, 256)   # どの時刻のトークンか
+ 空間位置エンコーディング (196, 256)  # どの空間位置のトークンか
```

V-JEPA の 1024 次元を 256 次元に圧縮しつつ、「このトークンは映像のどの時刻・どの場所に対応するか」という位置情報を付与する。Slot Attention が時空間構造を活用できるようにする前処理。

#### 3.2.2 SpatiotemporalSlotAttention

Slot Attention は「**物体を発見する**」ための機構である。1568 個のトークンを K=24 個のスロット（＝物体の候補）に競合的に振り分ける。

```
アルゴリズム:
1. 初期化: 各スロットを学習可能な平均 μ + ノイズ σ·ε で初期化
2. 3回のイテレーション:
   a) Q = f(slots), K = f(tokens), V = f(tokens)
   b) attention = softmax(Q·K^T / √d, dim=slot軸)
      └── スロット軸で softmax → スロット同士がトークンを奪い合う
   c) updates = 正規化された attention × V
   d) slots = GRU(updates, slots_prev)
   e) slots = slots + MLP(LayerNorm(slots))
```

核心は **softmax の軸がスロット方向**であること。通常の attention は query 方向に softmax をかけるが、Slot Attention ではスロット方向にかける。これにより:
- 各トークンは**最も関連性の高いスロットに割り振られる**
- スロット同士が**競合的にトークンを獲得**する（同じ物体の部分パッチが1つのスロットに集まる）
- 結果として、スロットが自然に物体単位の表現になる

GRU + 残差 MLP による反復更新が、この競合的割り当てを安定させる。

**なぜ Slot Attention か:**
- 微分可能（end-to-end 学習可能）
- 物体数が不定でも固定数のスロットで対応可能（未使用スロットは existence=0）
- 明示的な物体検出器（YOLO 等）が不要 → ドメイン非依存

#### 3.2.3 SlotRefinement

```
2層の TransformerEncoder (self-attention)
(B, 24, 256) → (B, 24, 256)
```

Slot Attention の出力に対してスロット間の self-attention を適用する。これにより:
- 重複するスロット割り当ての解消（同じ物体を複数スロットが捉えている場合の統合）
- スロット間の関係性エンコード（「人の手」と「人」が関連することを学習）

#### 3.2.4 TemporalTrajectoryExtractor

attention マップを使って、各スロット（≒物体）の時間的な軌跡を抽出する。

```
入力:
  slots:     (B, 24, 256)    ... スロット表現
  attn_maps: (B, 24, 1568)   ... 各スロットがどのトークンに注目しているか
  tokens:    (B, 1568, 256)  ... 元のトークン列

処理:
  1. attn_maps を (B, 24, 8, 196) に reshape（8時刻 × 196空間）
  2. 各時刻で空間方向に重み付き平均 → (B, 24, 8, 256) = 各物体の各時刻での表現

出力:
  identity:   (B, 24, 256)   ... 物体の静的特徴（スロットそのもの）
  trajectory: (B, 24, 8, 256) ... 物体の時間的変化（8時刻分）
  existence:  (B, 24)         ... 物体の存在確率 [0, 1]
```

**なぜ identity と trajectory を分けるか:**
- identity: 「何の物体か」（カテゴリ分類に使用）
- trajectory: 「その物体が時間的にどう変化したか」（イベント検出に使用）
- 後段の Event Decoder では `identity + mean(trajectory)` として結合し、静的・動的両方の情報を活用

#### ObjectRepresentation（出力データ構造）

```python
@dataclass
class ObjectRepresentation:
    identity:   Tensor  # (B, K, 256)    物体の静的表現
    trajectory: Tensor  # (B, K, 8, 256) 物体の時間軌跡
    existence:  Tensor  # (B, K)         存在確率
    categories: Tensor  # (B, K, 29)     カテゴリ分類ロジット (28カテゴリ + unknown)
    attn_maps:  Tensor  # (B, K, 1568)   Slot Attention のアテンションマップ
```

### 3.3 VJEPAEventDecoder（イベント予測）

発見されたオブジェクト群から「**イベント**（誰が何をしたか）」を予測するモジュール。DETR（DEtection TRansformer）の設計に倣い、固定数の**学習可能なイベントクエリ**を用いたセット予測を行う。

#### 3.3.1 Context Encoder

```
3層 TransformerEncoder (self-attention)
(B, K=24, 256) → (B, K, 256)
```

オブジェクトスロット間で self-attention を適用し、物体間の文脈をエンコードする。例えば「手がカップの近くにある」「人が椅子に座っている」といった空間的・意味的な関係性を捉える。

**なぜ ObjectPoolingModule の SlotRefinement と別にもう一段あるのか:**
- SlotRefinement は「物体発見の精度向上」が目的（Slot Attention の後処理）
- Context Encoder は「イベント予測のための文脈構築」が目的（Event Decoder の前処理）
- 役割が異なるため分離されている

#### 3.3.2 Event Decoder (Cross-Attention)

```
4層 TransformerDecoder (cross-attention)

Query:     (B, M=20, 256) ... 学習可能なイベントクエリ
Key/Value: (B, K=24, 256) ... Context Encoder 出力（オブジェクトスロット）
出力:      (B, M=20, 256) ... 各イベントクエリの表現
```

M=20 個のイベントクエリが、K=24 個のオブジェクトスロットに cross-attention をかけてイベント情報を収集する。各クエリは独立に動作し、それぞれが1つのイベントを担当する可能性がある。

**なぜ cross-attention か:**
- 各イベントクエリが**必要なオブジェクトだけに注目**できる（「take_out」イベントなら agent と target に注目）
- クエリ同士は self-attention も行うため、イベント間の整合性も保たれる
- 固定数のクエリで可変数のイベントを扱える（不要なクエリは interaction=0 と学習される）

### 3.4 予測ヘッド

Event Decoder の出力 (B, M, 256) に対して 7 つの独立した MLP ヘッドを適用し、イベントの各属性を予測する。

```
各ヘッド = Linear(256, 256) → ReLU → Dropout(0.1) → Linear(256, 出力次元)
```

| ヘッド | 出力形状 | 予測内容 | 損失関数 |
|---|---|---|---|
| **interaction** | (B,M,1) | このスロットはイベントか？ | BCE |
| **action** | (B,M,13) | アクション種別 | CE |
| **agent_ptr** | (B,M,K) | 行為者（どのオブジェクト？） | CE |
| **target_ptr** | (B,M,K) | 対象物（どのオブジェクト？） | CE |
| **source_ptr** | (B,M,K+1) | 取り出し元（+ "なし"） | CE |
| **dest_ptr** | (B,M,K+1) | 格納先（+ "なし"） | CE |
| **frame** | (B,M,16) | 発生フレーム | CE |

#### 13種類のアクション

| アクション | 説明 | source | dest |
|---|---|---|---|
| take_out | 容器から取り出す | 必要 | - |
| put_in | 容器に入れる | - | 必要 |
| place_on | 面の上に置く | - | 必要 |
| pick_up | 面から持ち上げる | 必要 | - |
| hand_over | 人から人へ渡す | 必要 | 必要 |
| open | 開ける | - | - |
| close | 閉める | - | - |
| use | 道具を使う | - | - |
| move | 場所Aから場所Bへ移動 | 必要 | 必要 |
| attach | 取り付ける | - | 必要 |
| detach | 取り外す | 必要 | - |
| inspect | 確認する | - | - |
| no_event | イベントなし | - | - |

#### ポインタヘッドの仕組み

agent_ptr と target_ptr は K 次元のロジットを出力し、softmax + argmax でどのオブジェクトスロットを指すかを決定する。無効なスロット（existence が低い）には `-inf` をマスクして選択されないようにする。

source_ptr と dest_ptr は K+1 次元で、最後のスロットが "なし" を表す。`open` や `close` のように source/dest が不要なアクションでは、この "なし" スロットが正解となる。

## 4. SAM3 パイプライン（レガシー）

SAM 3（Segment Anything Model 3）を使った旧パイプライン。V-JEPA パイプラインとは入力が異なるが、Event Decoder の設計思想は共通。

```
動画
 │
 ├── SAM 3 Tracker [凍結]
 │     └── 各フレームの物体追跡結果
 │           ├── object_embeddings: (N, 256)     SAM の埋め込み
 │           ├── object_temporal:   (N, 16, 12)  幾何特徴量の時系列
 │           └── pairwise:          (N, N, 16, 7) ペアワイズ特徴量
 │
 └── EventDecoder
       ├── geo_proj:          幾何特徴量を 256 次元に射影
       ├── emb_proj:          SAM 埋め込みを 256 次元に射影
       ├── temporal_encoder:  3層 self-attention（オブジェクト間）
       ├── pair_proj + agg:   ペアワイズ特徴量を集約して加算
       ├── context_encoder:   3層 self-attention（文脈エンコード）
       ├── event_decoder:     4層 cross-attention（M=20 クエリ）
       └── 7つの予測ヘッド     （V-JEPA 版と同一構造）
```

**V-JEPA パイプラインとの主な違い:**

| 観点 | SAM3 パイプライン | V-JEPA パイプライン |
|---|---|---|
| オブジェクト検出 | SAM 3 Tracker（外部） | Slot Attention（学習） |
| 入力特徴量 | 3種混合（埋め込み + 幾何 + ペアワイズ） | V-JEPA トークン 1種 |
| オブジェクト数 | 可変（最大K=30にパディング） | 固定 K=24 スロット |
| 関係性モデル | 明示的ペアワイズ特徴量 | attention で暗黙的に学習 |
| End-to-End性 | オブジェクト検出は外部依存 | 映像特徴量から一貫して学習 |

## 5. 損失関数と学習

### 5.1 Hungarian Matching

セット予測の核心は、**予測と正解の最適な対応付け**である。M=20 個の予測スロットと N 個の正解イベント（N ≤ M）を 1 対 1 で対応付ける必要がある。

```
コスト行列 C (M × N):
  C[i,j] = -log P(action_j | pred_i)    ... アクション一致度
          + -log P(agent_j  | pred_i)    ... 行為者一致度
          + -log P(target_j | pred_i)    ... 対象物一致度
          + -log P(frame_j  | pred_i)    ... 時刻一致度

最適割当: scipy.optimize.linear_sum_assignment(C)
```

Hungarian Algorithm により O(M³) で最適な 1 対 1 マッチングを求める。マッチされた予測スロットには interaction=1（イベントあり）、未マッチには interaction=0（イベントなし）が割り当てられる。

**なぜ Hungarian Matching か:**
- 予測の順番に依存しない（順列不変）
- 最適な割当を保証（貪欲法よりも安定）
- DETR で実証された手法であり、セット予測の標準的アプローチ

### 5.2 EventGraphLoss

Hungarian Matching 後に各ヘッドの損失を計算する。

```
total_loss = 2.0 × interaction_loss    (全 M スロット、BCE)
           + 1.0 × action_loss         (マッチのみ、CE)
           + 1.0 × agent_ptr_loss      (マッチのみ、CE)
           + 1.0 × target_ptr_loss     (マッチのみ、CE)
           + 0.5 × source_ptr_loss     (マッチのみ、CE)
           + 0.5 × dest_ptr_loss       (マッチのみ、CE)
           + 0.5 × frame_loss          (マッチのみ、CE)
```

- **interaction 損失**は全 M スロットに適用（イベントの有無判定）
- **その他の損失**はマッチしたスロットのみに適用（イベントの内容予測）
- interaction の重みが 2.0 と最も高いのは、イベント検出が最重要タスクであるため
- source/dest/frame の重みが 0.5 と低いのは、アクション種別によっては不要（optional）なため

### 5.3 VJEPAEventGraphLoss（2段階マッチング）

V-JEPA パイプラインでは、Slot Attention で発見したオブジェクトと正解オブジェクトの対応付けが追加で必要になる。

```
Stage 1: Slot-Object Matching（オブジェクトの対応付け）
════════════════════════════════════════════════════════
  コスト行列 (K=24 slots × N_obj objects):
    cost[k,n] = -log P(slot_k のカテゴリ = cat_n) + 0.5 × (-log existence[k])

  → Hungarian Matching でスロットと正解オブジェクトを対応付け
  → 正解イベントのオブジェクトインデックスをスロットインデックスに変換（リマップ）

  追加損失:
    category_loss:  マッチしたスロットのカテゴリ分類 CE
    existence_loss: 全スロットの存在判定 BCE

Stage 2: Event Matching（イベントの対応付け）
════════════════════════════════════════════════════════
  → リマップ済み正解イベントで EventGraphLoss を計算（5.2 と同一）
```

**なぜ 2 段階か:**
- Slot Attention は「どのスロットがどの物体か」を教師なしで発見するが、正解との対応は不明
- カテゴリベースの Hungarian Matching で対応を確立してから、イベント損失を計算する必要がある
- category_loss と existence_loss がオブジェクト発見の品質を制御する補助損失として機能

## 6. 学習時と推論時の違い

| 観点 | 学習時 | 推論時 |
|---|---|---|
| Slot 初期化 | μ + σ·ε（ノイズあり） | μ のみ（決定的） |
| 勾配計算 | あり（AMP float16） | なし（`torch.no_grad()`） |
| object_mask | matched slots ∪ (existence > 0.5) | existence > 0.5 |
| イベント判定 | Hungarian Matching で割当 | interaction > 0.5 でフィルタ |
| 出力 | 損失値 | argmax でクラス・ポインタ確定 |
| Sliding Window | - | clip_stride でオーバーラップ + NMS |

学習時の Slot Attention でノイズを加えるのは、スロット初期化の多様性を確保し、局所解への収束を防ぐため。推論時は決定的に動作させることで再現性を保証する。

## 7. 設計判断の背景

### なぜテキスト生成ではなくセット予測か

LLM/VLM でイベントを自然言語として生成する方法も考えられるが、以下の理由からセット予測を採用:

1. **構造保証**: セット予測は常に正しい構造の出力を生成する（ポインタは必ずオブジェクトを指す）。テキスト生成ではハルシネーションや不正な形式が発生しうる
2. **速度**: Event Decoder は 1 回の forward pass で全イベントを一括出力。自己回帰デコードより高速
3. **軽量**: VLM は学習データ生成にのみ使用し、推論時は Event Decoder（~10-15M パラメータ）のみ

### なぜ Slot Attention で物体を発見するか

1. **End-to-End 学習**: V-JEPA 特徴量から直接物体を発見し、イベント予測まで一貫して最適化
2. **ドメイン非依存**: 物体検出器の事前学習データに依存しない。カテゴリ語彙の変更にも柔軟
3. **可変物体数への対応**: 固定 K 個のスロットで任意数の物体を表現（未使用スロットは existence→0）

### なぜ Event Query が M=20 個か

1クリップ（16フレーム、約15秒）内のイベント数は通常 0~5 個程度。M=20 は十分な余裕を持たせた設計であり、未使用クエリは interaction=0 として学習される。多すぎても学習は安定するが、少なすぎるとイベントを取りこぼす。

### なぜ 7 つの予測ヘッドが独立しているか

各ヘッドは独立した MLP であり、相互に依存しない。これは:
- 各属性の予測難易度が異なるため、独立に最適化する方が安定
- ヘッドの追加・削除が容易（モジュラー設計）
- 損失の重みを属性ごとに調整可能

## 8. やさしい解説（一般向け）

### このシステムは何をするの？

動画を見て、「**誰が、何を、どうしたか**」を自動で読み取るシステムです。

例えば、オフィスの動画を入力すると:
- 「人がカップを棚から取り出した」
- 「人がノートPCを開いた」
- 「人がペンを机の上に置いた」

といった出来事を、コンピュータが理解できる形式（JSON）で出力します。

### どうやって動いているの？

人間が動画を見て「誰が何をしたか」を理解するプロセスを、3つのステップに分解しています:

#### ステップ 1: 動画を「見る」（V-JEPA）

まず、動画の各シーンを細かいパッチ（小さなタイル）に分割し、それぞれの意味を数値化します。これは人間が動画を見て「ここに何かある」と認識する過程に相当します。この部分は大量の動画で事前学習済みのモデル（V-JEPA）をそのまま使います。

#### ステップ 2: 「物体」を見つける（Slot Attention）

ステップ 1 で得られた大量のパッチ情報から、「これは人」「これはカップ」「これは棚」といった物体単位のまとまりを見つけます。

イメージとしては、ジグソーパズルのピースを「このピース群は空の絵、このピース群は山の絵」とグループ分けするような処理です。Slot Attention という仕組みを使い、24 個の「スロット」（入れ物）が競い合ってパッチを獲得することで、自然と物体ごとのグループができます。

#### ステップ 3: 「出来事」を読み取る（Event Decoder）

見つかった物体の情報を元に、「どんな出来事が起きたか」を予測します。20 個の「質問カード」（イベントクエリ）が物体情報を参照して、それぞれ「こんなイベントがあったのでは？」と回答します。

各回答には以下の情報が含まれます:
- **そもそもイベントがあったか？**（20枚中、実際にイベントを報告するのは数枚だけ）
- **何をした？**（「取り出した」「開けた」「置いた」など 13 種類から選択）
- **誰が？**（24個の物体スロットから選択）
- **何に対して？**（同上）
- **どこから／どこへ？**（移動を伴う場合）
- **いつ？**（動画クリップ内のどの時点か）

### なぜこの方法がよいの？

- **速い**: 動画1クリップあたり1回の計算で全イベントを出力。ChatGPT のように1文字ずつ生成する方式よりはるかに高速
- **正確な形式**: 出力は必ず決まった形式のデータ構造。「カップを取り出した...多分」のような曖昧な出力は発生しない
- **軽量**: 動画を理解する大きなモデル（V-JEPA）は事前計算済みで、実際に学習・推論するのは小さなモデル（Event Decoder）だけ

### 料理に例えると

1. **V-JEPA** = 農家（食材を育てる）: 動画から基本的な映像特徴量を抽出。大変な作業だが事前に済ませておく
2. **Slot Attention** = 仕分け担当（食材を分類する）: 大量の特徴量を「これは人参」「これは玉ねぎ」と物体ごとに整理
3. **Event Decoder** = シェフ（料理を作る）: 整理された食材（物体情報）を見て、「何の料理ができるか」（何のイベントが起きたか）を判断して盛り付ける（構造化データとして出力する）

この「分業」により、シェフ（Event Decoder）は食材を育てるところから始める必要がなく、整理された情報だけを見て素早く正確に判断を下せます。
